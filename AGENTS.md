# AGENTS.md

## О проекте

EkoSufity.pl - одностраничный сайт на Next.js для услуг по монтажу натяжных потолков во Вроцлаве и в радиусе до 100 км.

Текущая структура проекта компактная:

- `app/layout.tsx` - корневой layout и metadata.
- `app/page.tsx` - основная страница лендинга.
- `app/globals.css` - глобальные стили.
- `components/Header.tsx` - фиксированный верхний бар с прозрачным состоянием на первом экране и полупрозрачным белым фоном после прокрутки.
- `components/PriceCalculator.tsx` - калькулятор стоимости.
- `components/ContactForm.tsx` - форма контакта.
- `components/FAQAccordion.tsx` - FAQ.
- `components/FloatingActions.tsx` - плавающие CTA-элементы.

## Стек проекта

- Next.js 14 (`App Router`)
- React 18
- TypeScript
- Tailwind CSS 3
- PostCSS + Autoprefixer
- Lucide React для иконок

## Ключевые команды

Установка зависимостей:

```bash
npm install
```

Режим разработки:

```bash
npm run dev
```

Продакшн-сборка:

```bash
npm run build
```

Запуск production-сервера после сборки:

```bash
npm run start
```

Команда lint присутствует в `package.json`:

```bash
npm run lint
```

Но в текущем состоянии проекта ESLint не доведен до рабочего non-interactive режима: `next lint` может запрашивать первичную настройку. Поэтому основная техническая проверка сейчас - это `npm run build`.

## Правила кода

- Делать небольшие, локальные изменения без лишних рефакторингов.
- Сохранять текущую архитектуру: один основной экран в `app/page.tsx`, логика вынесена только в понятные UI-компоненты.
- Перед созданием новых компонентов сначала проверять, можно ли переиспользовать существующие.
- Не менять польский текст на странице без явного запроса.
- Не удалять существующие секции лендинга без явного запроса.
- Использовать существующие токены и палитру из Tailwind-конфига:
  - `ink`
  - `muted`
  - `porcelain`
  - `line`
  - `gold`
  - `gold-dark`
  - `charcoal`
- Для иконок использовать `lucide-react`, а не добавлять новые икон-пакеты без необходимости.
- Сохранять текущий визуальный характер интерфейса: премиальный, светлый, с аккуратными золотыми акцентами.
- Особенно аккуратно вносить изменения в hero-блок и верхний бар: это самая чувствительная часть страницы.
- Если правка касается header/hero:
  - учитывать фиксированное положение верхнего бара;
  - не ломать прозрачное состояние на первом экране;
  - не ломать переход к полупрозрачному белому фону при прокрутке.
- Предпочитать Tailwind-классы и существующий стиль проекта вместо добавления новой кастомной CSS-логики без необходимости.

## Правило ExecPlan

Для сложных задач, новых заметных фич и существенных рефакторингов сначала составлять `ExecPlan` по правилам из `PLANS.md`, а затем вести реализацию по этому плану.

`ExecPlan` нужен, если изменение:

- затрагивает несколько компонентов, секций или пользовательских сценариев;
- меняет поведение интерфейса в нескольких состояниях;
- требует серии зависимых шагов;
- содержит существенные неизвестные, риски или варианты реализации;
- может повлиять на адаптивность, навигацию, сборку или ключевые CTA.

Для мелких визуальных правок, точечных текстовых изменений и локальных исправлений `ExecPlan` не нужен.

Если `ExecPlan` создаётся или обновляется:

- он должен быть самодостаточным и не полагаться на прошлые планы для обязательного контекста;
- весь критичный контекст, решения и инструкции должны быть повторены в текущем плане;
- разделы `Progress`, `Surprises & Discoveries`, `Decision Log` и `Outcomes & Retrospective` нужно поддерживать актуальными;
- коммиты создавать только если этого требует workflow проекта или пользователь явно попросил.

## Правила вёрстки и UI

- Проверять desktop и mobile состояние хотя бы визуально.
- Не допускать наложения текста на ключевые CTA.
- Сохранять читаемость текста поверх изображений и градиентов.
- Не ломать якорную навигацию (`#home`, `#alex`, `#benefits`, `#cennik`, `#realizacje`, `#opinie`, `#faq`, `#kontakt`).
- Любые изменения в первом экране должны оставлять верхнее меню визуально связанным с hero-изображением.
- Не превращать секции в новые декоративные карточки, если текущая секция уже работает в существующем ритме страницы.

## Самопроверка перед завершением

После изменений нужно пройтись по этому чеклисту:

1. Запустить сборку:

```bash
npm run build
```

2. Убедиться, что сборка проходит без ошибок TypeScript и без падения Next.js.

3. Если менялась UI-часть, проверить страницу локально:

```bash
npm run start
```

или, если нужен режим разработки:

```bash
npm run dev
```

4. Проверить вручную:

- страница открывается;
- hero отображается корректно;
- header закреплен сверху;
- на первом экране header выглядит прозрачным;
- после прокрутки header получает ожидаемый полупрозрачный белый фон;
- мобильное меню открывается и закрывается;
- основные CTA работают;
- якорные ссылки прокручивают к нужным секциям.

5. Если после перезапуска сервера сайт отдает `500`, `404` на `/_next/...` или отображается как сырой HTML:

- считать это проблемой артефактов сборки;
- пересобрать проект заново;
- при необходимости очистить или переименовать старую папку `.next` и повторить `npm run build`.

## Что считать хорошим изменением

Хорошее изменение для этого проекта:

- не увеличивает сложность без пользы;
- не ломает лендинг на мобильных;
- не расходится с текущим визуальным стилем;
- проходит через `npm run build`;
- не требует дополнительных зависимостей без веской причины.
